# Conversion Plan: Go + Vanilla JS в†’ C#/.NET

## Current Architecture Summary

### Backend (`tools/`)
- **Language:** Go 1.24
- **Database:** SQLite via `modernc.org/sqlite` (pure Go driver)
- **CLI:** Custom command parser in `cmd/main.go` (no framework)
- **Structure:**
  - `cmd/main.go` вЂ” CLI entry point with command routing (454 lines)
  - `internal/db/db.go` вЂ” Data models, CRUD, CSV import, migrations (545 lines)
  - `internal/db/db_test.go` вЂ” 54 tests (1383 lines)
  - `internal/export/export.go` вЂ” JSON export to `site/data/` (40 lines)
  - `internal/export/export_test.go` вЂ” 11 tests (354 lines)
  - `db/migrate/001_init.sql` вЂ” Schema: tracks, events, event_classes, event_class_rules
  - `db/migrate/002_add_event_dates.sql` вЂ” Adds end_date column
- **Build:** Makefile with targets: test, build, init, seed, export, deploy, etc.
- **Tests:** 54 backend tests, в‰Ґ80% coverage required

### Frontend (`site/`)
- **Type:** Static HTML/CSS/JS (no framework)
- **Pages:** `index.html`, `events.html`, `event.html`, `about.html`, `about-technical.html`, `404.html`
- **JS Modules:** `app.js` (443 lines), `filters.js`, `year.js`
- **CSS:** Custom CSS + Bootstrap 5 via CDN (with SRI hashes)
- **Data:** Fetches `data/events.json` and `data/tracks.json` at runtime
- **Tests:** Vitest + Happy DOM, 92 tests, ~88% coverage
- **Dev Server:** `server.js` (native Node.js), `npm start`

### Deployment
- AWS S3 static hosting + CloudFront CDN
- `deploy.ps1` PowerShell script syncs `site/` to S3
- The Go CLI exports SQLite в†’ JSON в†’ `site/data/`

### Database Schema (4 tables)
```
tracks (id, name, city, address, url)
events (id, title, track_id, event_datetime, end_date, event_driver_fee, event_spectator_fee, url, description)
event_classes (id, event_id, name, buyin_fee)
event_class_rules (id, event_class_id, rule)
```

## Target Architecture: C#/.NET

### Technology Choices

| Component | Current | Target |
|-----------|---------|--------|
| Backend language | Go 1.24 | C# / .NET 9 |
| CLI framework | Custom switch/case | `System.CommandLine` |
| Database | SQLite (modernc.org/sqlite) | SQLite (`Microsoft.Data.Sqlite`) |
| ORM/Data | Raw SQL queries | Dapper (lightweight micro-ORM) |
| Migrations | Raw .sql files read by Go | Raw .sql files read by C# (same pattern) |
| JSON export | `encoding/json` | `System.Text.Json` |
| CSV import | `encoding/csv` | `CsvHelper` |
| Test framework | `go test` | xUnit + FluentAssertions |
| Coverage | `go tool cover` | Coverlet |
| Build system | Makefile | Makefile (calling `dotnet` CLI) |
| Frontend | Vanilla HTML/CSS/JS | **No change** (keep as-is) |
| Frontend tests | Vitest + Happy DOM | **No change** (keep as-is) |
| Dev server | `site/server.js` (Node.js) | **No change** (keep as-is) |
| Deployment | PowerShell в†’ S3 | **No change** (keep as-is) |

### Key Decisions
1. **Keep the frontend completely unchanged** вЂ” it's static HTML/JS fetching JSON files. No reason to touch it.
2. **Keep frontend tests unchanged** вЂ” Vitest/Node.js stays in `site/`.
3. **Keep deployment scripts unchanged** вЂ” `deploy.ps1` stays in `tools/aws/`.
4. **Use Dapper instead of EF Core** вЂ” the current Go code uses raw SQL. Dapper is the closest C# equivalent (lightweight, SQL-first). EF Core would be a huge architectural shift.
5. **Keep SQLite** вЂ” no reason to change the database engine.
6. **Keep the same CLI command structure** вЂ” all existing `make` targets should work identically.
7. **Keep the same JSON output format** вЂ” the frontend depends on the exact JSON structure.
8. **Solution file at repo root** вЂ” enables opening the full project in Visual Studio with all files visible (site, docs, tools) via Solution Folders.

## Risk Mitigation

### JSON Compatibility (HIGH RISK)
The frontend reads `events.json` and `tracks.json` directly. Field names and structure must be **exactly identical** after conversion.

**Mitigation:**
- Save current JSON output as golden files before conversion
- Write a comparison test that diffs .NET output against golden files
- Key field mappings to verify:
  - `start_date` (from Go's `StartDate` with `json:"start_date"`)
  - `end_date` (nullable, omit when null)
  - `event_driver_fee` / `event_spectator_fee` (nullable, omit when null)
  - `track_name` (from JOIN)
  - `classes` в†’ `rules` nesting (omit when empty)
  - `buyin_fee` (nullable, omit when null)

### DateTime Handling (MEDIUM RISK)
Go parses multiple SQLite datetime formats. C# must handle the same formats.

**Mitigation:**
- Port all 10 datetime format strings from Go to C# `DateTime.TryParseExact()`
- Test with actual data from the production database

### SQLite Driver Differences (LOW RISK)
Go uses `modernc.org/sqlite` (pure Go). C# uses `Microsoft.Data.Sqlite` (native).

**Mitigation:**
- Both drivers support the same SQL dialect
- The same `.sql` migration files will work with both
- Test with a copy of the production database

### CSV Import Parity (LOW RISK)
Go's `encoding/csv` and C#'s `CsvHelper` have slightly different defaults.

**Mitigation:**
- Configure CsvHelper to match Go's CSV reader behavior
- Port all CSV error-case tests
